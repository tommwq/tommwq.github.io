Java内存区域与内存溢出异常

Java虚拟机管理的内存划分为以下几个部分：

  • 方法区 Method Area
  • 堆 Heap
  • 虚拟机栈 VM Stack
  • 本地方法栈 Native Method Stack
  • 程序计数器 Program Counter Register

虚拟机栈、本地方法栈和程序计数器是线程私有的，方法区和堆是线程共享的。

方法区也叫做永久代 Permanent Generation，存储已经加载的类的信息和编译后的代码。方法区中，保存常量的部分叫做运行时常量池（Runtime Constant Pool）。运行期常量池可以在运行期修改，比如String.intern()。

对象的创建。虚拟机遇到new指令后，在方法区中检查类，必要时，加载、解析并初始化类。然后为新对象分配内存，新分配的内存中，每个字节都是0。之后进行对象的初始化，首先将元信息写入对象头（object header）。

对象在内存中的布局分为对象头、实例数据（instance data）和对齐填充（padding）。

垃圾收集器与内存分配策略

在判断一个对象是否可以回收时，Java将对象根据连通性（引用关系）划分为若干个互不相连的连通图。对于那些不包含GC根的连通图，其中的对象可以回收。这个过程叫做可达性分析。GC根不止一个、虚拟机栈中的对象、方法区中的静态成员和常量、JNI引用的对象都是GC根。

方法区空间也会被回收。方法区中的常量，如果没有被引用，可以被回收。方法区中的类，如果满足

  • 类的实例全都被回收。
  • 类的ClassLoader被回收。
  • 类的java.lang.Class对象没有被引用。

则也会被回收。

GC根主要是全局性引用和上下文引用。在进可达性分析时首先要扫描GC根，这个过程会导致应用线程停顿。即使对于CMS，枚举GC根也会发生STW。为了降低枚举GC根对应用线程产生的影响，JVM引入了安全点（safe point）的概念，只有当线程执行到安全点时才进行GC。安全点的选择以“具有让程序长时间执行的特征”为原则，通常会选择方法调用、循环跳转、异常跳转等地方。 如果由于线程调度的原因（比如阻塞），某个线程一致无法执行到安全点，这会导致GC无法进行。针对这种情况，JVM对安全点进行了扩展，引入了安全区域（safe region）的概念。在安全区域中，对象的引用关系没有发生变更，可以安全的进行GC。应用线程进入安全区域时，为自己设置一个标志位。JVM发起GC时，可以不用阻塞在安全区域中运行的线程。但是当线程将要离开安全区域时，它必须等待直到确认系统已经完成了GC根的枚举。

Serial收集器

Serial是一个单线程的垃圾收集器，只有一个线程负责GC，适用于桌面环境，或单CPU、内存较少的机器。

ParNew收集器

ParNew是Serial的多线程版本。在新生代GC时，ParNew采用多线程、STW和复制的方式回收对象。在老年代GC时，ParNew采用单线程、STW和标记-清理的方式回收。

Parallel Scavenge

Parallel Scavenge是新生代收集器，也是多线程、STW和复制的。Parallel Scavenge收集器会采用启发式算法，使得吞吐量（用户代码时间/(用户代码时间+GC时间)）尽量维持在用户期望（-XX:GCTimeRatio）的水平，或将GC停顿时间维持在用户期望（-XX:MaxGCPauseMillis）水平。

Serial Old收集器

Serial Old是Serial的老年代版本，采用标记-整理算法。

Parallel Old

Parallel Old是Parallel Scavenge的老年代版本，采用多线程的标记-整理算法。

CMS

Concurrent Mark Sweep的目标是缩短GC停顿时间。CMS在进行GC时执行以下4个步骤：

  • 初始标记。STW，标记GC根直接引用的对象。
  • 并发标记。
  • 重新标记。STW，修正并发标记期间，用户线程改变的对象引用。
  • 并发清除

CMS的缺点是对CPU要求较高，并且没有整理节点，容易产生内存碎片。并且发生Concurrent Mode Failure时，CMS会使用Serial Old进行老年代的回收。

G1

garbage-first。G1将堆划分为多个大小相同的区域（region），跟踪每个区域里的垃圾，并以此将区域保存在优先队列中。G1的垃圾回收分为以下几个步骤：

  • 初始标记
  • 并发标记
  • 最终标记
  • 筛选回收

虚拟机类加载机制

从加载到卸载的过程中，类的生命周期如下：

  • 加载 loading
  • 验证 verification
  • 准备 preparation
  • 解析 resolution
  • 初始化 initialization
  • 使用 using
  • 卸载 unloading

类的加载分为

  • 通过类的全限定名得到二进制字节流。
  • 将字节流转换为方法区的运行时数据结构。
  • 在内存中生成一个代表这个类的java.lang.Class对象。

准备阶段将对象赋予初始值。初始化阶段执行类的static初始化语句。

启动类加载器是C++实现的，包含在JVM中。其他的类加载器继承自java.lang.ClassLoader。

虚拟机字节码执行引擎

栈帧Stack Frame是用于支持虚拟机进行方法调用和执行的数据结构，保存了句柄变量表、操作数栈、动态连接和返回地址信息等。 

package org.fenixsoft.polymorphic;
/**
 *方法静态分派演示
 *@author zzm
 */
public class StaticDispatch{
    static abstract class Human{
    }
    static class Man extends Human{
    }
    static class Woman extends Human{
    }
    public void sayHello(Human guy){
        System.out.println("hello,guy！");
    }
    public void sayHello(Man guy){
        System.out.println("hello,gentleman！");
    }
    public void sayHello(Woman guy){
        System.out.println("hello,lady！");
    }
    public static void main(String[]args){
        Human man=new Man();
        Human woman=new Woman();
        StaticDispatch sr=new StaticDispatch();
        sr.sayHello(man);
        sr.sayHello(woman);
    }
}

运行结果： 

hello,guy！
hello,guy！

在上面的例子中，man变量的静态类型（static type，也叫外观类型apparent type）是Human，实际类型（actual type）是Man。在函数重载时，JVM根据静态类型判断，这个是编译期确定的。这个方法叫做静态分派。

MethodHandle是在字节码的层面，可以支持多种语言。

java字节码是基于栈的。以计算1+1为例，基于栈的指令类似于 

iconst_1 // 将常数1压栈
iconst_1 // 将常数1压栈
iadd     // 将操作数弹栈并计算，将结果压栈
istore_0 // 将结果保存到slot[0]中

而基于寄存器的指令类似于 

mov eax, 1
add eax, 1

基于栈的指令集更容易移植，缺点是执行速度较慢。

早期（编译期）优化

在C#中，从变异后的Intermediate Language的角度看，List<int>和List<String>是2个不同的类型，这种方式叫做类型膨胀，或真实泛型。Java则不同，在编译后，二者是相同的类型，泛型信息被擦除，只是在使用时，自动增加了类型转换。 实际上擦除的是Code属性中的字节码。在元数据中，还保留了泛型信息。

Java内存模型与线程

线程调度有协同式（cooperative）和抢占式（preemptive）。协同式调度是线程在执行完工作后，主动通知调度器（yield），让出CPU资源。抢占式调度是调度器抢占CPU，分配的其他线程。

Java线程有5种状态：

  • new
  • runnable
  • waiting。object.wait() thread.join locksupport.park()
  • timed waiting。thread.sleep()、object.wait(timeout)、thread.join(timeout)、locksupport.parkNanos()、locksupport.parkUtil()
  • blocked。
  • terminated。
