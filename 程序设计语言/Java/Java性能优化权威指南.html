操作系统性能监控

typeperf可以监控Windows系统性能： 

typeperf "\Processor(_Total)\% Privileges Time" "\Processor(_Total)\% User Time"

或者将参数写入文件，由typeperf加载： 

typeperf -cf cpu-util.txt

Linux上可以使用vmstat、mpstat和top等。

查看CPU任务队列

typeperf "\System\Processor Queue Length"

vmstat
uptime
top

查看内存使用率

typeperf -si 5 "\Memory\Available Mbytes" "\Memory\Pages/sec"

vmstat
free

JVM概览

HotSpot VM有3个主要的组件：VM运行时（Runtime）、JIT编译器（JIT Compiler）和内存管理器（Memory Manager）。

HotSpot VM的启动过程如下：

  • 解析命令行。
  • 设置堆大小和JIT编译器。
  • 设定环境变量，如LDLIBRARYPATH、CLASSPATH等。
  • 如果有-jar参数，从jar中读取Main-Class。否则从命令行读取Main-Class。
  • 使用JNI方法JNICreateJavaVM建立HotSpot VM。
  • 加载Main-Class。
  • 通过JNI方法CallStaticVoidMethod调用Java程序的main方法，启动Java程序。

类加载过程

JVM根据类或接口的名字寻找二进制数据，定义Java类，建立java.lang.Class对象。如果没有找到二进制数据，会抛出NoClassDefFound异常。加载时会检查类的格式，如果出现错误，会抛出ClassFormatError或UnsupportedClassVersionError。加载类之前会加载超类。如果继承层次出错，JVM抛出ClassCircularityError。如果类引用的接口实际上不是接口，JVM抛出IncompatibleClassChangeError。

类的加载由类加载器完成。类加载器可以委派其他加载器加载类。JVM中有3中类加载器：启动类加载器负责加载BOOTCLASSPATH中的类，如rt.jar。扩展类加载器负责lib/ext中的包，系统类加载器加载应用程序。

HotSpot VM中的线程有

  • VM线程
  • 周期任务线程
  • 垃圾收集线程
  • JIT编译线程
  • 信号分发线程

如果JVM发生致命错误，会生成文件hserrpid<PID>.log记录信息。

HotSpot VM使用分代垃圾回收机制，这个机制依赖于弱分代假设（weak generational hypothesis），该假设如下：

  • 大部分对象的生存周期较短。
  • 长生存周期对象很少引用短生存周期对象。

基于这个假设，HotSpot VM将内存划分为新生代、老生代和永久代。新生代存放新分配的对象，老生代存放长生存周期对象，永久代存放元数据，如类的数据结构等。在回收新生代对象时，为了提高效率，避免扫描老年代对象，HotSpot VM将老年代对象对新生代的引用记录在卡表（card table）里，并通过内存屏障同步卡表更新。为了避免对象过早晋升到老年代，新生代划分为3个区域：Eden和2个Survivor。Eden保存新分配对象，Survivor保存短生命周期对象。为了提高对象分配效率，JVM在Eden中划分一块区域作为TLAB（Thread Local Allocation Buffer线程本地分配缓冲区），避免分配对象时发生竞争。HotSpot VM提供了多种垃圾回收算法，Serial适合于对停顿时间要求不高的客户端程序。Throughput（Parallel）适合于批处理任务。CMS适合于任务突发的低延迟系统，如Web服务器。G1是为替代CMS开发的新的垃圾回收器。

JVM性能监控

关于垃圾收集的重要数据有：

  • 当前使用的垃圾收集器
  • Java堆大小
  • 新生代和老年代代销
  • 永久代大小
  • Minor GC持续时间
  • Minor GC的频率
  • Minor GC的空间回收量
  • Full GC的持续时间
  • Full GC的频率
  • 每个并发垃圾收集器周期内的空间回收量
  • 垃圾收集前后Java堆的占用量
  • 垃圾收集前后新生代和老年代的占用量
  • 垃圾收集前后永久代的占用量
  • 是否老年代后永久代的占用触发了Full GC
  • 应用是否显示的调用了System.gc()

HotSpot VM报告垃圾收集数据几乎没有额外开销，可以在生产环境中使用。

如果目标JVM配置了JMX： 

com.sun.management.jmxremote.port=<port>
com.sun.management.jmxremote.ssl=<true|false>
com.sun.management.jmxremote.authenticate=<true|false>

可以通过JMC远程生成堆栈信息。

使用-XX:+PrintCompilation可以观察JIT编译器的行为。
