分支

仓库拥有master分支、develop分支两个长期分支和feature、release、hotfix三类短期分支。

master分支

master分支用来记录发布历史，master分支以“master”命名。

develop分支

develop分支用于集成和记录开发历史，develop分支以“develop”命名，基于master分支建立。

feature分支

feature分钟用于记录尚未完成开发工作的新功能。feature分支以“feature-XXX”为命名格式，其中XXX可以是特性编号或特性名称。feature分支从develop分支建立，当开发任务完成后，feature分支合并到develop分支，然后删除。

release分支

release分支用于准备发布版本。release分支以“release-XXX”命名，其中XXX是版本号。当发布条件成熟时，从develop分支建立release分支。release分支不再接受新功能，只接受缺陷修复和发布相关的提交。当版本冻结后，release分支合并到master分支，并建立标签，标签名为版本号。然后将release分支合并到develop分支，接着删除release分支。

hotfix分支

hotfix分支用于处理紧急修复情况。hotfix分支以“hotfix-XXX”命名，其中XXX是缺陷编号或缺陷名称。hotfix分支从master分支建立。在完成紧急修复后，hotfix分支合并到master分支，并建立标签。然后将hotfix分支合并到develop分支，接着删除hotfix分支。

提交信息

提交信息应当清楚说明变更内容，并保持简洁。

提交信息格式

提交信息应当采用下列格式： 

type(scope): subject

body

footer

其中type是变更类型，scope是变更影响范围，subject是主题，body是变更描述，footer是注脚。

变更类型、变更影响范围和主题位于同一行，这一行也叫做提交信息头。提交信息头、变更描述和注脚之间，分别用一个空白行分隔。

变更类型

变更可以是下列类型之一

类型 说明
feat 添加新特性
fix 修复缺陷
docs 编辑文档
style 调整代码格式
refactor 重构软件
test 测试相关变更
chore 辅助工具变更

变更影响范围

变更影响范围由系统模块划分决定。对于通常的web项目，可以分为view、controller、model。

主题

主题应当简单明了的描述变更，应当采用英文编写，首字母小写，以动词开头，使用第一人称现在时，结尾不加句号（.），且长度不超过50字符。

变更描述

变更描述要充分简洁的说明变更的目的和内容，采用英文编写，使用第一人称现在时。内容较多时可以拆分成多行，每行不超过70字符。

注脚

注脚的目的有两个，一是声明不兼容变更，二是声明关闭issue。如果变更没有涉及这两部分，可以省略注脚和注脚前的空行。当声明不兼容变更时，注脚应当以“BREAKING CHANGE”开头，并描述变更影响。当声明关闭issue时，注脚应当采用下列格式： 

Closes #XXX

或 

Closes #XXX, #YYY, #ZZZ

这里的XXX、YYY和ZZZ均为issue编号。如果一个提交同时包含不兼容变革和issue关闭，在编写注脚时，将不兼容变更声明放在issue关闭声明之前。

Revert提价

当需要撤销某个提交时，提交信息的格式为 

revert: MMM

This reverts commit XXX.

其中MMM是待撤销提交的提价信息头，XXX是待撤销提交的commit值。
